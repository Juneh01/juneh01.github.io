AICAS 2026 PPU Track - Qwen3-VL Inference Optimization

Project Overview

This project focused on fixed-weight inference optimization for Qwen3-VL-2B-Instruct on the Alibaba PPU-ZW810E platform for the AICAS 2026 challenge. The target was not to train a new model, edit the checkpoint, or upload quantized weights, but to keep the pretrained model fixed and optimize the execution path around it.

The final submission achieved:

Metric Value
Composite score 2.3803
Accuracy 0.7178
TTFT 19.78 ms
Throughput 369.80 tokens/s

The work sits at the intersection of VLM serving, accelerator-aware kernel optimization, runtime profiling, quantized inference layouts, and system-level performance engineering.

Task and Constraints

The competition evaluated a vision-language model with three practical objectives:

  • preserve answer accuracy under hidden evaluation data;
  • reduce time-to-first-token (TTFT);
  • increase autoregressive decode throughput.

The finals ran on PPU machines. The observed platform exposed a CUDA-compatible programming surface, but native extensions had to be rebuilt for the PPU environment. The active machine configuration was roughly:

Component Observation
Accelerator PPU-ZW810E
Compute units 64 CUs
HBM 96 GB
Reported L2 / LLC 64 MB
Runtime stack PyTorch 2.8, CUDA 12.9-compatible PPU SDK

The key constraints shaped the solution: no training, no batch-size change, no shipped modified checkpoint, and no accuracy-destroying shortcut that only optimizes one metric.

Baseline

On 150 local PPU samples, the baseline recorded about 78.53 ms TTFT and 58.99 tokens/s throughput. The main overhead came from visual prefill, text prefill, Python/control-plane scheduling, repeated kernel launches, KV-cache movement, and per-token decode fragmentation.

Profile-Driven Workflow

I organized optimization around a repeatable profile loop:

This workflow helped separate real bottlenecks from local illusions. A candidate was not promoted only because one kernel became faster; it also had to preserve answer behavior and improve the end-to-end metrics.

The profile stack served different roles:

  • ASYS identified end-to-end kernel, API, synchronization, and memory-copy cost.
  • ACU inspected occupancy, registers, scheduler stalls, cache hit rate, and memory behavior for a specific kernel.
  • SM profiler traces decomposed the persistent decode kernel into phases such as QKV projection, QK/RoPE/KV write, attention, OProj/PostNorm/MLP, grid synchronization, and final normalization.
  • Answer comparison guarded against speedups that collapsed generated answers.

Main Optimization Ideas

Visual and Prefill Path

The prefill path was optimized conservatively. Instead of relying on aggressive visual-token deletion, the system used bounded image budgets, direct resize policies, positional-cache reuse, and vision/text big-operator fusion where it was safe.

For text prefill, packed QKV and packed gate/up operations reduced fragmented pointwise and reshaping overhead. KV-cache preparation was also aligned with the custom decode path so that prefill outputs could be consumed with fewer copies.

Persistent Mega Kernel Decode

The largest throughput gain came from replacing the standard generate loop with a persistent decode path. Instead of launching many kernels for every layer and every generated token, the Mega Kernel executed the main transformer decode pipeline on device:

This moved the bottleneck from framework orchestration into a smaller number of profileable kernels. That was useful: once the cost concentrated in the decode kernel, I could reason about register pressure, synchronization, shared memory, memory bandwidth, and phase-level hotspots directly.

Load-Time W8 Auxiliary Layouts

The final system constructed auxiliary W8 layouts at model load time for selected hot decode projections. This kept the source checkpoint unchanged while allowing the fused kernel to read compact INT8 weights and per-row scales in bandwidth-sensitive matvec loops.

The important distinction is that this was an inference layout optimization, not a newly trained or shipped quantized model. The active W8 paths were designed to be branchless inside hot loops, avoiding the performance cost of runtime FP16/W8 branches.

Synchronization and Dataflow Cleanup

Later-stage work focused on small but meaningful decode-kernel cleanups:

  • removed unnecessary bootstrap grid synchronization;
  • reduced full-grid barrier overhead where safe;
  • staged the MLP intermediate in a compact shared-memory page for the down projection;
  • trimmed stale arguments and inactive fallback branches from the decode call graph;
  • specialized the active MLP path so dead pointers and dead branches did not perturb register allocation and code generation.

These changes were not glamorous, but they mattered after the major fusion work was already done. At that stage, unused arguments, register pressure, and shared-memory layout could still affect throughput.

The early phase moved the implementation from a framework-heavy path to a PPU-compatible optimized runtime. The middle phase established the persistent decode kernel and safe image/prefill policies. The final phase improved the already-fused path through load-time W8 layouts, TTFT KV reuse, MLP dataflow cleanup, synchronization trimming, and call-graph simplification.

Engineering Takeaways

This project reinforced a few lessons:

  • A fast kernel is not enough; the full inference path includes preprocessing, prefill, decode, cache layout, launch overhead, synchronization, and output policy.
  • Profile tools must be used together. Timeline, counter profile, phase trace, benchmark JSON, and answer comparison each catch a different failure mode.
  • Fixed-weight optimization still leaves a large design space: runtime dispatch, auxiliary layouts, kernel fusion, cache reuse, and memory staging can move performance substantially without changing model weights.
  • VLM optimization is harder than text-only LLM serving because visual token policy affects both TTFT and answer quality.
  • The final 1-2% often comes from removing boring overhead: dead arguments, inactive branches, unnecessary barriers, and poor shared-memory staging.
