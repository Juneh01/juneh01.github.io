Team Design Project - STM32 与 OpenMV 小车项目

项目概览

Team Design Project 主要围绕嵌入式小车系统展开，项目材料包含 STM32 主控、OpenMV 视觉识别、UART 通信、电机与编码器、ST-LINK、RGB 检测代码和移动机器人相关参考资料。

这个项目是典型的软硬件结合实践：上层要做视觉检测和目标识别，下层要控制电机运动，中间还要通过串口通信把感知结果传给主控。

硬件组成

项目涉及的硬件模块包括：

  • STM32 主控板。
  • OpenMV 摄像头模块。
  • 编码器电机。
  • TB6612 等电机驱动。
  • ST-LINK 下载调试工具。
  • 小车底盘与机械结构。

这些模块之间的集成比单独写代码复杂得多。硬件接线、电源、电机方向、编码器反馈、串口波特率和调试工具都可能成为系统运行的瓶颈。

OpenMV 视觉识别

OpenMV 端主要负责颜色识别和目标检测。材料中包含 RGBdetecting.py、single_color_rgb565_blob_tracking_1.py、TDPS.py、uart.py 等脚本。

视觉部分的典型流程：

  1. 从摄像头读取图像。
  2. 将图像转换到适合颜色阈值检测的格式。
  3. 根据 RGB565 或颜色阈值检测目标 blob。
  4. 计算目标位置、面积或偏移量。
  5. 通过 UART 将检测结果发送给 STM32。

这部分让我熟悉了嵌入式视觉中的一个现实问题：算法不能只看识别准确率，还要考虑帧率、光照变化和通信延迟。

STM32 主控与通信

STM32 端负责接收 OpenMV 数据，并根据目标位置控制电机。材料中包含 main.c、openmv.c、openmv.h、main.h 等代码文件。

系统中的关键点包括：

  • UART 数据帧格式设计。
  • 接收缓冲区与解析逻辑。
  • 电机 PWM 控制。
  • 编码器反馈。
  • PID 或简单闭环控制。

如果视觉检测结果不稳定，控制端也会出现抖动。因此感知和控制需要一起调试。

工程调试经验

这个项目的难点并不在某个单一算法，而在系统集成：

  • OpenMV 识别到的坐标是否稳定。
  • 串口数据是否丢包或格式错位。
  • 电机方向和速度是否符合预期。
  • 编码器反馈是否正确。
  • 主控循环频率是否能跟上视觉结果。

这些问题让我更深刻地理解了“能运行”和“能稳定运行”之间的区别。

我的收获

TDPS 项目训练了我从硬件、底层驱动、通信协议到视觉脚本的完整调试能力。它不像纯软件项目那样可以只靠日志定位问题，很多时候需要同时看代码、串口、示波器/调试器和实际运动效果。
