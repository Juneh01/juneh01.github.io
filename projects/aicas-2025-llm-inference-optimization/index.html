IEEE AICAS 2025 Grand Challenge - 大模型软硬系统优化

项目概览

IEEE AICAS 2025 Grand Challenge 的“大模型软硬系统优化”赛道关注大语言模型在资源受限硬件上的高效推理。这个项目的核心问题并不是简单地把模型跑起来，而是在有限内存、有限算力和严格延迟约束下，让模型尽可能稳定、快速、低成本地完成推理。

我整理和实践的重点集中在端侧 LLM 推理优化：低比特权重量化、查找表计算、矩阵乘法分块、CPU 指令优化、prefill/decode 阶段瓶颈分析，以及 MNN、T-MAC、llama.cpp 等推理框架的部署思路。

背景问题

大模型部署到边缘设备时，主要瓶颈来自三个方面：

  • 计算开销：FP16/FP32 推理依赖大量矩阵乘法，CPU 上传统 GEMM 很难取得足够吞吐。
  • 内存占用：即使是 0.5B 级别模型，未压缩参数和 KV cache 也会给内存带来压力。
  • 功耗与延迟：端侧场景不能只追求峰值性能，还要考虑稳定延迟和能耗。

因此，项目目标可以概括为：在不显著牺牲精度的前提下，通过量化和计算路径重排降低模型体积、减少内存访问，并提升 CPU/端侧硬件上的推理吞吐。

技术路线

低比特量化

项目中重点关注 4-bit 权重量化路径。低比特量化可以显著降低模型参数体积，减少内存带宽压力。围绕量化配置，我记录了 bits、group size、activation setting 等参数对推理行为的影响。

量化带来的收益主要有两类：

  • 模型参数体积降低，端侧加载更容易。
  • 线性层计算可以转化为更硬件友好的低比特计算流程。

量化的风险也很明显：过低比特会带来困惑度上升、输出质量下降，甚至在某些任务中出现明显退化。因此项目文档里没有只记录速度，也同时关注精度和 PPL 变化。

T-MAC 查找表计算

T-MAC 的思路是把低比特权重拆解成多个 1-bit 矩阵，再按若干 bit 分组，对可能出现的 bit pattern 进行查找表预计算。在线推理时，部分乘加操作可以替换成查表与累加。

这个过程包含两个阶段：

  • Offline stage：对权重进行分解、分组、重排，生成计算所需的 LUT。
  • Online stage：输入激活后，通过 LUT 快速组合结果，减少传统 GEMM 中重复的乘加成本。

这类方法的关键不是“是否做了量化”这么简单，而是量化后计算如何映射到内存布局、缓存和指令集。项目中因此同时关注 tiling、连续内存布局、缓存命中和不同 CPU 指令路径。

MNN 端侧推理优化

在 workshop 笔记中，我整理了 MNN 做端侧 LLM 推理优化时的几条主线：

  • 权重量化与 QAT。
  • DiskEmbedding 降低内存压力。
  • 半精度与内存复用。
  • prefill 与 decode 阶段的不同瓶颈。
  • 线性层中的 W4A8 量化、loop tiling 和指令选择。
  • attention 层中的 F16 矩阵乘法、分块和 fused MHA kernel。

这些内容帮助我把 LLM 推理拆成几个可分析的硬件瓶颈，而不是把“模型很慢”当成一个笼统问题。

实验与记录

项目材料中记录了 Qwen 系列小模型的推理优化路线，以及原始 FP16 与 4-bit 优化版本在吞吐、内存占用和首 token 延迟上的对比。实验报告中还包含 llama.cpp batched-bench、困惑度评估、长上下文测试等验证方向。

这些实验让我形成了一个比较清晰的判断：端侧大模型优化不是单点技巧，而是一组协同设计问题。模型精度、量化粒度、内存布局、batch 策略、线程绑定和硬件指令都互相影响。

我的收获

这个项目把我从“使用大模型”推到了“理解大模型怎么在硬件上运行”的层面。最有价值的部分是学会用工程指标拆解问题：prefill 为什么慢，decode 为什么受内存访问影响，低比特为什么不等于自动加速，以及为什么同一个模型在不同 backend 上会表现出截然不同的推理特征。

后续如果继续深化，我会重点补充真实设备 profiling、不同量化方法的系统对比，以及更清楚的端到端部署复现实验。
