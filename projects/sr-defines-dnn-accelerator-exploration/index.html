SR - DeFiNES DNN Accelerator 调度空间探索

项目概览

DeFiNES 是一个面向 DNN accelerator 的 design space exploration 框架，全称为 Enabling Fast Exploration of the Depth-first Scheduling Space for DNN Accelerators through Analytical Modeling。我在 SR 阶段整理了论文、源码和复现实验说明，重点理解 depth-first scheduling 对能耗和延迟的影响。

这个方向把神经网络计算映射到硬件架构和调度策略上，关注的不再是某个网络能否训练，而是一个 workload 如何在 accelerator 上更高效执行。

背景问题

DNN accelerator 的性能不仅取决于硬件结构，也取决于调度方式。对于卷积神经网络等 workload，不同层之间、不同 tile 之间的数据复用方式会直接影响片上存储、DRAM 访问、能耗和延迟。

传统 layer-by-layer scheduling 会逐层执行网络，而 depth-first scheduling 尝试跨层融合更多计算，在局部区域内连续处理多个层，以提升数据复用。

但 depth-first scheduling 的设计空间很大，包括：

  • tile size。
  • fuse depth。
  • X/Y 方向划分。
  • workload 类型。
  • accelerator 架构参数。

DeFiNES 的价值就在于用分析模型快速探索这些组合，而不是完全依赖耗时仿真。

工具链理解

DeFiNES 的复现材料中包含：

  • workload 输入。
  • hardware architecture 输入。
  • depth-first scheduling 参数。
  • 自动运行实验脚本。
  • 结果 pickle 文件。
  • plot 脚本和可视化图。

官方 artifact 中以 FSRCNN 在 Meta-prototype-like depth-first architecture 上的调度探索为例，遍历多种 depth-first scheduling options，并输出能耗和延迟对比图。

关键学习点

Scheduling 是一等设计变量

做神经网络加速时，模型结构、硬件结构和调度策略必须一起考虑。同一个 workload 在不同调度方式下，DRAM 访问和片上复用会有很大差别。

能耗与延迟并不总是一致

某些调度可能降低能耗，但不一定带来最低延迟。设计空间探索的意义在于找到 Pareto front，而不是只看单一指标。

分析模型能加快早期探索

完整硬件仿真通常很慢，而分析模型可以在早期快速筛选设计点，为后续更精细仿真或实现缩小范围。

我的收获

DeFiNES 让我第一次从“模型部署”进一步看到“模型调度和硬件映射”。这和后续 AICAS 端侧大模型优化有明显联系：无论是 CNN accelerator 还是 LLM inference，最终都绕不开数据移动、缓存、tile、并行度和硬件瓶颈。

如果继续完善这篇项目记录，我会补充自己跑出的结果图、不同 tile 参数的对比，以及和 ZigZag 等 DSE 工具的关系。
